# 📐 Шаблон проектного CLAUDE.md (Hard Rules + Session-Closing Pipeline)

> Каркас для `CLAUDE.md` **внутри конкретного проекта** (не глобального `~/.claude/CLAUDE.md`).
> Источник паттерна: r/ClaudeCode «Advanced Claude Code workflow» (90/100, автор выпустил realtime-карточную игру Film Cue на SvelteKit/Cloudflare/PartyKit).
> Закрывает два пробела в стеке NEXUS: (1) проектные жёсткие правила с причинами, (2) единый ритуал закрытия сессии.
>
> **Как использовать:** скопировать блоки ниже в `<project_root>/CLAUDE.md`. Глобальный NEXUS-CLAUDE.md остаётся как есть — этот файл его дополняет на уровне проекта.

---

## Почему это работает (мета-паттерн)

- **Каждое правило с причиной** → правило не выбрасывают при следующей чистке и не нарушают «по незнанию». «Однажды убил на это целый день» — лучшая защита от повтора.
- **Жёсткие правила в начале файла** → приоритет над остальными инструкциями (как §1 в глобальном CLAUDE.md).
- **Тесные циклы обратной связи** → Claude сам ловит регрессии за один shell-вызов, не дожидаясь пользователя.
- **Проверка в реальном рендере** → убирает ~80% «выглядит правильно, но не работает».

## Блок 1 — вставить в начало проектного CLAUDE.md

```markdown
# CLAUDE.md — {Название проекта}

## ⛔ Hard Rules (приоритет над всем остальным)

> Каждое правило = ограничение + ПРИЧИНА. Без причины правило не добавляем.
> Пополняется по ходу проекта: поймали грабли → сразу фиксируем сюда с «почему».

### Окружение и сборка
- **{Правило}** — Причина: {что сломалось/почему так}. Пример: «Node version pinned 20.x — Причина: PartyKit падает на 22 при сборке.»
- **{Команды запуска/сборки}** — Причина: {нюанс}. Пример: «Билд только через `npm run build:cf`, не `vite build` — Причина: нужен Cloudflare adapter, иначе 500 на проде.»

### API и библиотеки
- **Проверять имена API/методов по живой документации (WebFetch) ПЕРЕД применением советов ревью** — Причина: модель галлюцинирует сигнатуры; правка по памяти ломает билд.
- **{Версия/нюанс библиотеки}** — Причина: {breaking change / deprecation}.

### Безопасность / анти-чит (если применимо)
- **{Правило фильтрации состояния}** — Причина: предотвратить утечку (напр. «сервер никогда не шлёт чужие карты в state — Причина: видны в devtools, чит в мультиплеере»).

### UI / визуал
- **UI-задача НЕ считается готовой без скриншота Playwright в mobile + dark theme** — Причина: ~80% «выглядит правильно» на деле сломано на мобиле/тёмной теме.

## 🔁 Feedback Loops (Claude гоняет сам)

Настроить В НАЧАЛЕ проекта, не потом:
- **Unit:** `{команда}` — напр. `npm run test` (Vitest), быстрый прогон.
- **E2E / UI:** `{команда}` — напр. `npx playwright test`.
- **Один вызов «всё сразу»:** `{команда}` — объединить unit+typecheck+lint в одну shell-команду, чтобы Claude запускал за раз. Напр. `npm run check` = `vitest run && tsc --noEmit && eslint .`.
- **Скриншот UI:** `{команда/скрипт Playwright}` — captures mobile + dark, кладёт в `./screenshots/`.

## ✅ Session-Closing Pipeline (ритуал конца сессии)

Перед тем как объявить работу завершённой — пройти по порядку:
1. **Review** — `/code-review` (или `/security-review` если трогали auth/данные).
2. **Simplify pass** — `/simplify` → убрать абстракции, не нужные для задачи (ловит over-engineering).
3. **Build + typecheck** — прогнать `{команда check}`, дождаться зелёного.
4. **UI grounding** (если менялся UI) — скриншот в реальном рендере, СМОТРЕТЬ вывод, чинить что видно, перезапустить.
5. **Scoped commit** — атомарный коммит с {emoji-конвенция проекта}. Напр.: ✨ feat · 🐛 fix · ♻️ refactor · ✅ test · 📝 docs.
```

## Блок 2 — emoji-конвенция коммитов (опционально, под проект)

```
✨ feat     — новая фича
🐛 fix      — багфикс
♻️ refactor — рефактор без смены поведения
✅ test     — тесты
📝 docs     — документация
⚡ perf     — производительность
🔒 security — безопасность
🎨 ui       — визуал/стили
🔧 chore    — конфиги/сборка
```

## Привязка к существующему стеку NEXUS + Fablize

| Шаг workflow из поста | Чем закрыто у тебя |
|---|---|
| UI-проверка скриншотом в реальном рендере | Fablize `verification-grounding-pack` (grounding loop) |
| Simplify pass | скилл `/simplify` |
| Code review | `/code-review`, `/review`, `/security-review` |
| Verify перед «готово» | `/verify` + Fablize финальный гейт (§10.2) |
| Субагенты для параллельной рутины | NEXUS §4 (параллельный dispatch) |
| Расследование багов | Fablize `investigation-protocol` |
| **Hard Rules с причинами** | ← новое, этот шаблон |
| **Единый Session-Closing ритуал** | ← новое, этот шаблон |

**Что НЕ дублируем:** механику тестов/grounding/review — она уже есть как скиллы и Fablize-паки. Этот шаблон только (1) даёт каркас проектных правил и (2) связывает существующие инструменты в один конвейер закрытия сессии.
