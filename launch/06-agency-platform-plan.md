# 06 — NEXUS: AI-агентство с отделами и оркестрацией. Детальный план

> Превращение текущего чата (marketing-office) из плоского списка 32 маркетинговых
> агентов в полноценное AI-агентство: отделы, подотделы, двухуровневая оркестрация,
> гибридный UX, чёрно-белый интерфейс, подписка. Источник новых агентов —
> `agency_agents/` (форк github.com/msitarzewski/agency-agents).

## Контекст и цель
Сейчас: юзер вручную выбирает одного из 32 агентов и общается один-на-один.
Нужно: юзер пишет задачу в чат → **планировщик** разбирает её, раздаёт подзадачи
нужным отделам → **оркестраторы отделов** распределяют между агентами → агенты
выполняют → результаты собираются обратно (агент → оркестратор отдела → общий
оркестратор → готовый ответ юзеру). Плюс возможность зайти в отдел и работать с
агентом напрямую (гибрид). 131 новый агент из `agency_agents` переводим на русский,
адаптируем под РФ и встраиваем в отделы — волнами, не трогая текущие 32.

**Решения владельца:** темп — каркас + волнами; режим — гибрид; подписка — низкая
фиксированная + freemium; интерфейс — чёрно-белый, выдвижной сайдбар отделов.

## 2. Модель данных агентов (как хранить иерархию)
- Каждый агент в `agents/<отдел>/<файл>.md`. Frontmatter дополняется двумя полями:
  `department` (к какому отделу) и `subdepartment` (вкладка внутри отдела).
- Текущим 32 агентам проставляется `department: Маркетинг` + их нынешняя `category`
  маппится в подотдел. **Тела и логика не трогаются** — только два поля в шапке.
- Загрузчик агентов строит дерево Отдел → Подотдел → Агенты. UI и API отдают эту
  иерархию вместо плоского списка.
- Каждый отдел описывается в едином файле-реестре (название, иконка, порядок,
  список подотделов, краткое описание для планировщика — по нему он понимает, за
  что отдел отвечает).

## 3. Двухуровневая оркестрация (главная фича)

### Поток «задача → результат»
1. **Юзер** пишет задачу в общий чат Оркестратора.
2. **Планировщик** (1 вызов модели) разбирает задачу: какие отделы нужны, что от
   каждого требуется, в каком порядке (что параллельно, что последовательно).
3. **Предпросмотр плана** показывается юзеру: «Задействую отделы: Инженерия (бэкенд),
   Дизайн (UX), Маркетинг (лендинг). Примерно N шагов / ~оценка запросов к модели».
   Юзер **подтверждает** (или правит/отменяет). Это ключевой контроль расхода токенов.
4. Для каждого отдела — **оркестратор отдела** (1 вызов модели): дробит подзадачу
   отдела на задания, выбирает нужных агентов отдела (1–N).
5. **Агенты** выполняют задания (могут параллельно), пользуясь веб-поиском и т.д.
6. **Сборка снизу вверх:** агенты → оркестратор отдела синтезирует результат отдела
   → общий оркестратор синтезирует **финальный ответ** юзеру (структурированный,
   с разбивкой по отделам и единым итогом).

### Передача между уровнями (handoff)
Переиспользуем формат NEXUS: **Контекст → Что нужно → Критерии готовности →
Доказательства**. Каждое задание агенту содержит эти 4 блока — это устраняет потерю
контекста (главная причина сбоев мультиагентных систем).

### Прогресс для юзера
Во время работы показывается **дерево статусов**: отдел → агенты, со статусами
(ждёт / работает / готово / ошибка). Юзер видит, кто что делает, а не «думает…».

### Контроль стоимости (ключ юзера — нельзя жечь бесконтрольно)
- **Предпросмотр плана с оценкой** числа вызовов модели до запуска (шаг 3).
- **Лимит глубины/ширины:** максимум отделов и агентов на задачу (настройка).
- **Быстрый режим:** для простого вопроса — один профильный агент без оркестрации
  (гибрид это позволяет). Оркестратор сам предлагает быстрый режим, если задача мелкая.
- **Retry ≤ 3** на агента, дальше — эскалация в ответе («не смог, причина»).
- Счётчик израсходованных токенов по итогу задачи виден юзеру.

## 4. Гибрид UX (как уживаются два режима)
- **Оркестратор** (звёздочка вверху сайдбара) — общий чат: кидаешь задачу, идёт
  распределение по отделам. Для комплексных задач.
- **Прямой заход в отдел** — клик по иконке отдела → попадаешь в отдел, видишь
  вкладки подотделов и агентов, выбираешь агента и общаешься один-на-один (как
  сейчас). Для точечной работы.
- Оба режима — один интерфейс, одна история переписки по контекстам.

## 5. Интерфейс и навигация (чёрно-белый, выдвижной сайдбар)
- **Узкая полоса слева** с иконками отделов (Lucide, монохром). Сверху — Оркестратор.
- **При наведении на полосу** — выезжает панель с названиями отделов (как референс
  21st.dev/1603). **При наведении на иконку** — тултип с названием отдела.
- **Клик по отделу** → область отдела: сверху вкладки подотделов, под ними — карточки
  агентов (как сейчас список в маркетинге).
- **Палитра — чёрно-белая:** убрать индиго (#6366f1), заменить на монохром (чёрный
  фон, белый/светло-серый текст и акценты, тонкие серые границы). Сохранить premium-
  ощущение (как сейчас, но без цвета).
- Ложится на текущий `index.html`: grid меняем на «полоса иконок + выезжающая панель
  + рабочая область»; рендер списка агентов переиспользуется внутри отдела.

## 6. Трансформация 131 агента (волнами)
**Конвейер** (расширение существующего build-подхода): для каждого агента из
`agency_agents/<dept>/*.md` — перевод тела на русский + адаптация под РФ (заменить
иностранные реалии/инструменты на российские аналоги где уместно, учесть РФ-законы),
приведение шапки к нашему формату (имя, эмодзи, русское описание, department,
subdepartment), запись в `agents/<отдел>/`. Существующие 32 — не трогаем.

**Приоритет волн:**
- **Волна 1** (вместе с каркасом): Инженерия, Дизайн, Продажи, Продукт и проекты —
  самые востребованные, не привязаны к региону, дают сразу «полное агентство».
- **Волна 2:** Аналитика и поддержка, Тестирование и QA, топ Спецотделов.
- **Волна 3:** остаток Спецотделов, Игры и XR, Наука и контент (длинный хвост).
- **Ещё не переведённые** агенты видны в структуре отдела как «скоро» (серые,
  неактивные) — чтобы был виден масштаб, но не ломалось.

## 7. Подписка (низкая фиксированная + freemium)
Токены везде — ключ юзера. Наша подписка — только за платформу и оркестрацию.

| Тариф | Цена | Что входит |
|-------|------|-----------|
| **Free** | 0 ₽ | Прямой чат с агентами базовых отделов (Маркетинг + 1–2). Без оркестратора. Свой ключ. |
| **Pro** | ~690–990 ₽/мес | Оркестратор (авто-распределение по отделам), **все** отделы, приоритет, история. |
| (опц.) **Team** | позже | Несколько мест, общие пресеты — на будущее. |

- Гейтинг — через наш `nexus-auth-server`: тариф зашит в подписке (план), при входе
  клиент знает, какие отделы и оркестратор доступны. Закрытые отделы видны, но с
  замком и предложением апгрейда (воронка freemium).
- «Не в тягость»: низкая фикс-цена + бесплатный вход — юзер уже платит за токены сам,
  психологически подписка воспринимается как доступ к инструменту, а не как «ещё один
  счёт за ИИ».

## 8. Фазы реализации

**Фаза 1 — Каркас (платформа отделов).**
Модель отделов/подотделов; реестр отделов с иконками; новый чёрно-белый интерфейс с
выдвижным сайдбаром и вкладками; текущие 32 агента раскладываются в отдел «Маркетинг»;
прямой чат по отделам работает. Оркестратора пока нет. → На выходе: то же, но в новой
навигации отделов, готово к наполнению.

**Фаза 2 — Волна 1 агентов.**
Конвейер трансформации; перевод и встройка Инженерии, Дизайна, Продаж, Продукта.
Отделы наполняются реальными агентами.

**Фаза 3 — Оркестратор (главная фича).**
Планировщик + предпросмотр плана + оркестраторы отделов + сборка снизу вверх + дерево
прогресса + контроль токенов и быстрый режим. Гибрид заработал полностью.

**Фаза 4 — Подписка и гейтинг.**
Тарифы Free/Pro в auth-сервере; замки на отделах/оркестраторе; экран апгрейда.

**Фаза 5 — Волны 2–3 и полировка.**
Остальные отделы; полировка интерфейса и копий; пересборка `.exe` с новым продуктом;
прогон Reality Checker перед релизом.
