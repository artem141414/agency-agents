# 05 — Модель дистрибуции и онлайн-авторизация (УТОЧНЁННАЯ, приоритетная)

> Это уточнение от владельца переопределяет приоритет из арт.01.
> Продукт №1 теперь — **self-hosted клиент с онлайн-авторизацией**, не Claude Code плагин.

## Суть модели (со слов владельца)
1. Юзер **скачивает наш репозиторий**, запускает по инструкции **на своём железе**.
2. **Авторизация онлайн** → проверка подписки на нашем сервере → юзер авторизован.
3. Юзер вставляет **свой Anthropic API-ключ** → токены за его счёт.
4. Работает даже на **4 ГБ RAM**.

## Почему это лучше, чем варианты из арт.01–04
| Проблема прошлых вариантов | Решение в этой модели |
|---|---|
| Плагин: нет DRM, промпты утекают (арт.04 риск №2) | **Онлайн-авторизация = реальный отзываемый контроль.** Нет подписки → клиент не стартует. Промпты на диске, но без логина бесполезны (нечем оркестрировать, обновления отрезаны). |
| SaaS: учёт токенов, дыра в монетизации (арт.02 риск №1) | **Ключ юзера → нет учёта токенов у нас.** Риск №1 исчезает. |
| SaaS: дорогая мультитенантная инфра (арт.03) | **Вычисления на железе юзера.** Наш сервер только авторизует — копеечная нагрузка. |
| SaaS: Anthropic API недоступен с РФ IP (риск №3) | **Это забота юзера** (его ключ, его сеть, его VPN/relay). |

## Архитектура: тонкий сервер + толстый клиент

```
┌─────────────────────────────────────┐         ┌──────────────────────────┐
│  КЛИЕНТ (репо, качает юзер)          │         │  НАШ СЕРВЕР (online)      │
│  marketing-office/ на железе юзера   │         │  nexus-auth-server/      │
│                                      │         │                          │
│  Flask app.py :5000 (локально)       │         │  Flask/уWSGI             │
│   ├─ экран ЛОГИНА ──────────────────────POST /api/auth/login──────────►  │
│   │                                  │◄────────  JWT (24–72ч) ───────────│
│   ├─ ввод API-КЛЮЧА юзера            │         │  проверка подписки в БД  │
│   │   → ~/.nexus/config.json (локал) │         │  (SQLite→Postgres)       │
│   ├─ Anthropic(client) = КЛЮЧ ЮЗЕРА  │         │                          │
│   ├─ /api/chat (gate: нужен JWT) ────────POST /api/auth/validate────────►│
│   │   → api.anthropic.com (ключ юзера)│        │  webhook ЮKassa/Gumroad  │
│   └─ 32+ агента локально (.md)        │◄───────  активация подписки ──────│
└─────────────────────────────────────┘         └──────────────────────────┘
   ВСЯ LLM-работа здесь, на ключе юзера            Только auth. ~50 МБ RAM,
   RAM: ~50–100 МБ (Flask+requests+bs4)            десятки тысяч юзеров на 1 VPS
```

**Граница:** наш сервер НЕ видит ни ключ юзера, ни его диалоги, ни токены. Только: «этот аккаунт оплачен — да/нет». Минимум данных = минимум ответственности (152-ФЗ/GDPR проще).

## Бюджет памяти 4 ГБ (анализ по факту app.py)
Локально тяжёлых вычислений НЕТ — LLM на стороне Anthropic. Потребление клиента:
- Python + Flask: ~25–40 МБ
- anthropic SDK + requests + bs4 + lxml: ~30–60 МБ
- 32 агента в памяти (тексты .md): ~1–2 МБ
- **Итого процесс: ~60–110 МБ.** На 4 ГБ — запас 30×.

Что НЕЛЬЗЯ тащить в self-hosted на 4 ГБ:
- ❌ Локальный LLM (Ollama/llama.cpp) — не нужен, используем API.
- ❌ Обязательный Docker Desktop (ест ~2 ГБ на Windows) → **основной путь установки нативный** (Python venv), Docker — опционально.
- ❌ Postgres/Redis на клиенте → локальный стор = JSON-файл (`~/.nexus/`), не БД.
- ⚠️ gunicorn с 1–2 воркерами (не `cpu*2+1`), либо встроенный Flask-сервер для локалхоста.

## Что меняется в клиенте (app.py) — план интеграции
1. **Ключ юзера вместо env** (строка 30 `client = Anthropic()`): глобальный `client` пересоздаётся из `~/.nexus/config.json`; если ключа нет → 412 «нужен ключ».
2. **Auth-gate**: декоратор `@require_auth` на `/api/chat` и `/api/agents`. Проверяет локальный JWT; протух → 401 → фронт показывает логин.
3. **Новые роуты клиента**:
   - `GET /` → если нет сессии, отдаёт экран логина; иначе чат.
   - `POST /api/auth/login` → проксирует на наш сервер, сохраняет JWT в `~/.nexus/session.json`.
   - `POST /api/config` → сохраняет API-ключ юзера локально (обфускация base64/keyring), пересоздаёт `client`.
   - `GET /api/status` → залогинен ли, есть ли ключ, до какой даты подписка.
4. **Периодическая ре-валидация**: при старте и раз в ~12ч — `POST /api/auth/validate`; подписка истекла → разлогин.
5. **Фикс usage** (app.py:537-541, опционально): суммировать токены всех итераций — чтобы показывать юзеру ЕГО расход (нам для биллинга не нужно).

## Наш сервер (nexus-auth-server) — состав
- `POST /api/auth/register` — создать аккаунт (в бою — через платёжку; для теста с trial 14 дней).
- `POST /api/auth/login` — email+пароль → JWT с claims (user_id, plan, sub_expires).
- `POST /api/auth/validate` — Bearer JWT → `{valid, plan, expires}` (проверка подписки в реальном времени).
- `POST /api/webhook/payment` — заглушка под ЮKassa/Gumroad: активирует/продлевает подписку.
- `GET /healthz` — для мониторинга.
- БД: SQLite (старт) → Postgres (рост). Пароли: werkzeug hash. Токены: PyJWT.
- Хостинг: Timeweb (арт.03), ~1 VPS тянет десятки тыс. аккаунтов (нагрузка только на логин).
